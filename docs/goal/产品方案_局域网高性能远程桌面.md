## 一、PM vs 架构师：核心决策辩论

### 辩论1：传输协议选型

| 维度 | PM 主张 | 架构师主张 | 结论 |
|------|---------|------------|------|
| 首选方案 | WebRTC（浏览器零安装） | 自定义UDP + FEC（Sunshine路线） | **双通道架构** |
| 理由 | 控制端无需安装客户端，手机/平板直接打开浏览器即可操控 | TCP 有队头阻塞，WebSocket 底层仍是 TCP；局域网虽低丢包，但 WiFi 下仍有抖动 | 视频流走 WebRTC(UDP)，控制信令走 WebSocket(可靠) |
| 妥协点 | 接受 WebRTC 需要 STUN 信令服务器的额外复杂度 | 接受 WebRTC 生态成熟度优势，放弃自造轮子 | LAN 场景无需 TURN 中继，STUN 即可打洞/直连 |

**最终裁决：WebRTC (aiortc) + WebSocket 双通道。**
- 视频/音频流：WebRTC UDP 通道，利用内置 FEC/NACK 机制
- 控制信令 + 文件传输 + 剪贴板：WebSocket 可靠通道
- LAN 场景下延迟可控制在 20-50ms 端到端

### 辩论2：屏幕捕获方案

| 维度 | PM 主张 | 架构师主张 | 结论 |
|------|---------|------------|------|
| 首选方案 | MSS（跨平台，简单） | DXcam（Desktop Duplication API） | **DXcam + MSS 降级** |
| 理由 | 希望一套代码覆盖 Win/Mac/Linux | MSS 基于 GDI，在 Win10+ 上性能远不如 DDA；DXcam 可达 240FPS 抓取 | Windows 主力用 DXcam，其他平台用 MSS |
| 风险 | DXcam 不捕获光标，不捕获 UWP/DRM 窗口 | 这是 Windows 安全策略限制，非库的问题。光标可用 RapidShot 或 WinRT 后端捕获 | 接受限制，标注在用户文档中 |

**最终裁决：DXcam 为主 (output_color="BGR")，可选叠加 cursor capture。**

### 辩论3：编码方案

| 维度 | PM 主张 | 架构师主张 | 结论 |
|------|---------|------------|------|
| 首选方案 | CPU 软编码（兼容性最好） | GPU 硬件编码（NVENC/AMF/QSV） | **GPU 优先 + CPU 兜底** |
| 理由 | 不是所有用户都有独显 | CPU 软编码 1080p@60fps 延迟 15-30ms，NVENC 仅 <5ms。且 CPU 编码会让宿主机的应用变卡 | 启动时自动检测 GPU 编码器，降级链：NVENC → AMF → QSV → libx264 |
| 风险 | - | FFmpeg 进程管理复杂，崩溃恢复需要额外处理 | 采用子进程 + 心跳监控模式 |

**最终裁决：FFmpeg 子进程硬件编码，H.264 基准，H.265/AV1 可选。**

### 辩论4：AI 超分增强

| 维度 | PM 主张 | 架构师主张 | 结论 |
|------|---------|------------|------|
| 观点 | 差异化卖点！低带宽下自动超分还原清晰文字 | 延迟太高，Real-ESRGAN 单帧 0.23s(RTX4090)，完全不可实时 | **Phase 2 离线增强，Phase 1 不做** |
| 妥协 | 接受 v1.0 不做实时 AI 超分 | 预留"画质增强"模式接口，未来 FlashVSR tiny 模式可能在 RTX 5060+ 跑到 30FPS | 架构上预留后处理管线 |

**最终裁决：v1.0 不做 AI 超分。保留后处理管线接口，Phase 2 评估 FlashVSR tiny / CuNNy。**

### 辩论5：客户端形态

| 维度 | PM 主张 | 架构师主张 | 结论 |
|------|---------|------------|------|
| 观点 | Web 前端优先，零安装 | Python GUI(PySide6) 可打包为 EXE，且能直接调用 DXcam 解码渲染 | **双客户端策略** |
| 策略 | Web 客户端用于快速接入 | Native 客户端用于高性能场景（游戏/设计） | Web 先行（v1.0），Native 跟进（v1.5） |

**最终裁决：v1.0 交付 Web 客户端（浏览器可直接解码 WebRTC 流），v1.5 交付 PySide6 原生客户端（支持硬件解码 + 更低延迟）。**

### 辩论6：网络发现机制

| 维度 | PM 主张 | 架构师主张 | 结论 |
|------|---------|------------|------|
| 观点 | 手动输入 IP 即可，局域网场景简单 | 必须做 mDNS 自动发现，否则体验太差 | **mDNS + 手动IP 双模式** |
| 实现 | 用户打开控制端，自动列出局域网内运行了受控端的机器 | python-zeroconf 库即可实现 | 默认自动发现，也支持手动输入 IP:Port |

**最终裁决：mDNS 自动发现优先，手动 IP 保底。**

## 二、产品定位与核心指标

### 产品定位
面向**局域网场景**的**高性能**远程桌面软件，对标 RDP(微软远程桌面) 和 VNC，差异化在于：
- **更低延迟**（20-50ms vs RDP 的 50-150ms）
- **更高帧率**（60FPS vs RDP 的 30FPS）
- **更轻量部署**（pip install 即可，无需系统配置）
- **Web 客户端**（浏览器直接访问，无需安装客户端）
- **开源可控**（基于 Python，无黑盒组件）

### 核心性能指标（KPI）

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 端到端延迟（有线） | < 30ms | 高速相机拍屏测时差 |
| 端到端延迟（WiFi 5G） | < 50ms | 同上 |
| 视频帧率 | 60 FPS（自适应） | statistics 面板 |
| 视频编码延迟 | < 5ms (NVENC) | FFmpeg bench |
| 屏幕捕获延迟 | < 3ms | DXcam timestamp |
| CPU 占用（受控端） | < 5% (GPU 编码时) | 任务管理器 |
| 内存占用（受控端） | < 200MB | 任务管理器 |
| 带宽占用（1080p 静态） | < 1 Mbps | 网络监控 |
| 带宽占用（1080p 动态） | < 20 Mbps | 网络监控 |
| 连接建立时间 | < 2s | 计时 |
| 断线重连 | < 3s 自动恢复 | 拔线测试 |

## 四、技术栈详细选型

### 4.1 Runtime & 包管理

| 组件 | 选型 | 版本要求 | 理由 |
|------|------|----------|------|
| Python | CPython | 3.11+ | asyncio 成熟，性能优于 3.10 |
| 包管理 | uv / pip | - | uv 更快，pip 兼容性更好 |
| 打包 | PyInstaller + Nuitka | - | Nuitka 编译为 C 后性能更好 |
| 虚拟环境 | venv | - | 标准库 |

### 4.2 屏幕捕获层

| 组件 | 选型 | 说明 |
|------|------|------|
| 主捕获 | **DXcam** (`dxcam[cv2]`) | Desktop Duplication API 封装，~240FPS @ 1080p |
| 光标捕获 | **RapidShot** | DXcam 不捕获光标，RapidShot 可获取光标位置/形状/可见性 |
| 降级捕获 | **MSS** (`python-mss`) | 当 DDA 不可用时（如虚拟机、旧系统）自动降级，~76FPS |
| 图像处理 | **OpenCV** (`opencv-python`) | numpy 格式、色彩空间转换、缩放 |
| 数值计算 | **NumPy** | 所有图像 buffer 的基础格式 |

**捕获管线：**
```
DXcam.grab() → numpy.ndarray (BGR, 原始分辨率)
  → cv2.resize() (可配置输出分辨率)
  → cv2.cvtColor() (BGR→YUV420p，匹配编码器输入)
  → FFmpeg stdin pipe
```

### 4.3 编码层

| 组件 | 选型 | 说明 |
|------|------|------|
| 编码引擎 | **FFmpeg** (子进程) | 最完整的硬件编码器支持 |
| Python 绑定 | **ffmpeg-python** | 声明式 API，自动构建命令行 |
| 一级编码器 | **h264_nvenc** (NVIDIA NVENC) | 延迟最低 <5ms，生态最成熟 |
| 二级编码器 | **h264_amf** (AMD AMF) | AMD 显卡用户 |
| 三级编码器 | **h264_qsv** (Intel QSV) | Intel 核显用户 |
| 四级降级 | **libx264** (CPU 软编码) | 无 GPU 或 GPU 编码器不可用时 |
| 高级编码 | **hevc_nvenc / av1_nvenc** | H.265/AV1，需高端 GPU，带宽更省 |

**编码参数策略（低延迟优先）：**
```
-preset p1           # NVENC 最低延迟预设
-tune ll             # 低延迟调优
-delay 0             # 零编码延迟
-rc cbr              # 恒定码率（流式场景首选）
-g 120               # GOP=2s (60fps×2)，控制关键帧间隔
-bf 0                # 禁用 B 帧，消除重排序延迟
-refs 1              # 单参考帧
```

### 4.4 传输层

| 组件 | 选型 | 说明 |
|------|------|------|
| 视频/音频流 | **WebRTC** (`aiortc`) | P2P UDP 传输，内置 NACK/FEC |
| 信令协议 | **WebSocket** (`websockets`) | SDP 交换 + ICE candidate 中继 |
| 控制通道 | **WebSocket** | 键鼠事件、剪贴板同步、文件传输协商 |
| 文件传输 | **WebSocket + 分片** | 大文件 64KB 分片，支持断点续传 |
| LAN 发现 | **mDNS** (`python-zeroconf`) | 自动发现局域网受控端 |
| 加密 | **DTLS-SRTP** (WebRTC 内置) | 传输加密。LAN 可选关闭加密以降低延迟 |

**端口规划：**
| 端口 | 用途 |
|------|------|
| 8765 | WebSocket 信令 + 控制通道 |
| 8766 | HTTP 静态服务 (Web 客户端) |
| 49152-65535 | WebRTC UDP 媒体流 (由 ICE 协商，动态分配) |
| 5353 | mDNS 广播 |

### 4.5 输入模拟层

| 组件 | 选型 | 说明 |
|------|------|------|
| 键盘模拟 | **PyNput** 或 **SendInput (ctypes)** | SendInput 更底层，延迟更低，兼容 DirectInput 游戏 |
| 鼠标模拟 | **SendInput** (ctypes 直接调 Win32 API) | 绝对坐标 + 相对坐标双模式 |
| 剪贴板 | **win32clipboard** (pywin32) | 文本/图片/文件剪贴板同步 |
| 手柄模拟 | **ViGEmBus** (可选) | 虚拟手柄驱动，支持 Xbox360/DS4 手柄 |

### 4.6 Web 客户端

| 组件 | 选型 | 说明 |
|------|------|------|
| 视频渲染 | `<video>` tag + WebRTC API | 浏览器原生解码 H.264/VP8/VP9 |
| 输入捕获 | DOM Events (keydown/keyup/mousemove/mouseup/mousedown/wheel) | 全屏锁定指针 API |
| UI 框架 | 原生 HTML/CSS/JS | 轻量无框架依赖 |
| 剪贴板 | `navigator.clipboard` API | 文本同步；图片需额外处理 |
| 触控支持 | Touch Events API | 手机/平板操控 |

## 五、核心模块设计

### 5.1 自适应质量引擎 (AQE)

这是产品体验的关键差异化模块。根据网络状况和画面内容动态调整编码参数。

**输入信号：**
- 网络 RTT（WebSocket ping 测量）
- 丢包率（WebRTC stats API）
- 可用带宽估计（WebRTC GCC 算法）
- 画面变化率（帧间像素差异百分比）
- 画面内容类型（文字/视频/游戏，基于变化区域特征判断）

**输出决策（5 档质量档位）：**

| 档位 | FPS | 分辨率 | 码率 | 触发条件 |
|------|-----|--------|------|----------|
| 极速 | 60 | 1080p | 20 Mbps | RTT<5ms, 有线, 画面剧烈变化 |
| 高速 | 60 | 1080p | 10 Mbps | RTT<10ms, WiFi 5G, 正常使用 |
| 标准 | 30 | 1080p | 5 Mbps | RTT<30ms, 办公场景 |
| 节能 | 30 | 720p | 2 Mbps | RTT<50ms, WiFi 2.4G |
| 极限 | 15 | 720p | 0.5 Mbps | RTT>50ms, 弱信号 |

**升降档策略：**
- 连续 3 秒满足高一级条件 → 升档
- 连续 1 秒触发低一级条件 → 降档（快速降档保护体验）
- 用户手动锁定档位时关闭自动调节

### 5.2 脏区域检测与增量传输

**目标：** 静态画面（办公/阅读场景）带宽降至接近零。

**算法：**
1. 维护前一帧的 numpy 数组
2. 计算 `cv2.absdiff(current, previous)` 得到差异图
3. `cv2.threshold(diff, threshold=15)` 二值化
4. `cv2.findContours` → 得到变化区域的 bounding boxes
5. 合并相邻 bbox（间距 < 16px）
6. 仅将变化区域送入编码器（通过 FFmpeg crop filter）
7. 编码后的区域帧携带领部坐标，控制端叠加渲染

**效果预估：**
- 静态码字场景：带宽降低 90%+
- 视频播放场景：脏区域接近全屏，退化到全帧传输
- 开销：差异检测约 0.5ms (1080p on numpy)

### 5.3 键鼠事件聚合与传输

**问题：** 高频鼠标移动事件（最高 1000Hz 游戏鼠标）如果每个事件都单独发送 WebSocket 消息，会导致网络风暴和 CPU 浪费。

**聚合策略：**
```
采集线程 (高频) → 环形缓冲区 → 编码线程 (每 5ms 聚合一批)
```

- 鼠标移动：合并为最新位置（中间位置丢弃）
- 鼠标点击：保留顺序，打时间戳
- 键盘按下/抬起：保留顺序，打时间戳
- 滚轮：累积 delta 值

每 5ms 打包一批事件发送，单个 WebSocket 消息含一批事件。LAN 5ms 间隔可保证 <10ms 输入延迟。

### 5.4 连接生命周期管理

```
┌─────────┐    ┌──────────┐    ┌──────────┐    ┌───────────┐
│  IDLE   │───→│ PAIRING  │───→│ ACTIVE   │───→│ RECONNECT │
└─────────┘    └──────────┘    └──────────┘    └───────────┘
     ▲                              │                  │
     └──────────────────────────────┘←─────────────────┘
               断开 / 超时
```

**状态机：**
- **IDLE**: 启动后等待连接。mDNS 广播在线状态。
- **PAIRING**: 首次连接需输入 PIN 码（4-6位数字），通过 WebSocket 传输。
- **ACTIVE**: 正常会话中。WebRTC 流 + WebSocket 控制双通道。
- **RECONNECT**: 网络闪断时，保留会话 token 60 秒。客户端自动重连，恢复 WebRTC 连接（ICE restart）和控制通道。

**安全模型：**
- PIN 码首次配对后，生成持久化 token 存在本地
- 后续连接使用 token 验证，无需重复输入 PIN
- 受控端可配置"每次询问"模式
- LAN 内可选关闭 DTLS 加密以降低延迟（默认开启）

### 5.5 多显示器支持

- 受控端枚举所有显示器（DXcam 支持 `output=0,1,2...`）
- 控制端可切换/组合显示
- 方案A：单显示器切换（默认）
- 方案B：多显示器拼接为一个大画面
- 方案C：多窗口分别显示不同显示器

## 八、风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| DXcam 在双显卡笔记本上捕获黑屏 | 高 | 中 | 自动检测并切换输出适配器；降级到 MSS |
| FFmpeg 子进程崩溃导致流中断 | 高 | 中 | 心跳监控 + 自动重启 FFmpeg + 关键帧快速恢复 |
| WebRTC 在某些企业防火墙下被拦截 | 中 | 低 | LAN 场景一般不经过防火墙；提供 TCP-only 降级模式 |
| aiortc 库维护不活跃 | 高 | 低 | 锁定版本 + Fork 自维护准备 |
| DXcam 不捕获光标 | 中 | 确定 | RapidShot 叠加光标；或 WinRT 后端 |
| UWP/DRM 窗口黑屏 | 低 | 确定 | 文档明确说明限制；Windows 安全策略无法绕过 |
| Python GIL 限制帧率 >160FPS | 低 | 低 | 目标 60FPS 完全在安全范围内 |
| PyInstaller 打包体积过大 | 中 | 高 | 裁剪无用模块，最终目标 < 80MB |
