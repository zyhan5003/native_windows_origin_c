# 局域网高性能远程桌面 — 模块方案逐项评审

> 版本: v1.0 | 日期: 2026-06-25 | 状态: 详细评审
> 本文档是对《产品方案》中每个模块的深入技术对比与选型论证。
> 每个模块提供 2~3 个可行方案，标注 **【选定】** 标识最终决策。

---

## M1: 屏幕捕获 (Screen Capture)

### 方案 A: DXcam — Desktop Duplication API (DDA)

**原理:** 通过 DXGI `IDXGIOutput1::DuplicateOutput()` 获取桌面帧。GPU 纹理 → CPU 内存映射 → numpy 数组。

| 维度 | 评估 |
|------|------|
| 捕获帧率 | ~240 FPS (1080p, 有变化时) |
| 捕获延迟 | 1-3ms (GPU 纹理直接映射) |
| CPU 占用 | 极低 (DMA 传输) |
| 光标捕获 | **不支持** (需另配 RapidShot/WinRT) |
| DRM 内容 | 黑屏 (Windows 安全策略，无法绕过) |
| 双显卡笔记本 | 可能捕获黑屏，需手动指定 GPU 适配器 |
| Python 绑定 | `dxcam[cv2]` — 成熟，维护活跃 |
| 输出格式 | numpy.ndarray (BGR/BGRA) |
| 窗口级捕获 | 不支持 (仅全显示器，然后裁剪) |

### 方案 B: Windows Graphics Capture (WGC) via zbl

**原理:** WinRT `Windows.Graphics.Capture` API。高层 API，由 OS 管理 GPU 资源。

| 维度 | 评估 |
|------|------|
| 捕获帧率 | 500-700 FPS (理论，受显示器刷新率约束) |
| 捕获延迟 | 0.8-1.8ms |
| CPU 占用 | 极低 |
| 光标捕获 | **内置支持** |
| DRM 内容 | 黑屏 |
| 双显卡笔记本 | OS 自动处理，无黑屏问题 |
| 窗口级捕获 | **原生支持** — 可只捕获指定窗口 |
| 遮挡窗口 | **正常工作** — 即使被遮挡也能捕获 |
| Python 绑定 | `zbl` (Rust+Python, PyPI 可用) |
| 系统要求 | Win10 1803+ (最佳 Win11 24H2) |
| 成熟度 | 较新，生态不如 DDA 成熟 |

### 方案 C: MSS — GDI 传统捕获

**原理:** Windows GDI `BitBlt`。最传统、兼容性最好。

| 维度 | 评估 |
|------|------|
| 捕获帧率 | 30-76 FPS |
| 捕获延迟 | 5-15ms |
| CPU 占用 | 中等 (需通过 GDI 管线) |
| 光标捕获 | 支持 |
| DRM 内容 | 黑屏 |
| 双显卡笔记本 | 无问题 |
| 跨平台 | Linux / macOS / Windows |
| Python 绑定 | `python-mss` — 非常成熟 |

---

### M1 辩论实录

> **PM:** "选 MSS 吧，跨平台，兼容性好，代码简单。万一将来要支持 Mac 和 Linux 呢？"
>
> **架构师:** "MSS 在 Windows 上走 GDI 管线，1080p 才 76FPS，而且每帧都要经过 GDI→DWM→CPU 的路径，延迟 5-15ms。DDA 直接走 GPU 显存 DMA，延迟不到 3ms，帧率差 3 倍。我们产品定位就是 Windows 高性能，不是跨平台。"
>
> **PM:** "那 WGC 呢？zbl 看起来帧率更高，还自带光标，还支持窗口捕获。"
>
> **架构师:** "WGC 确实是更现代的 API，但 zbl 库还很新，生态不够成熟。DDA 方案有 DXcam 这个已经在大规模使用的库，踩过的坑都修了。而且 DDA 在游戏/直播行业验证了多年。WGC 可以留作 Phase 2 演进方向。"
>
> **PM:** "双显卡笔记本黑屏问题怎么办？"
>
> **架构师:** "DXcam 支持 `output_device` 参数手动指定 GPU 适配器。启动时自动枚举所有适配器，选正确的那个。如果都失败，降级到 MSS 兜底。三层 fallback：DDA → WGC → MSS。"
>
> **PM:** "那光标呢？"
>
> **架构师:** "用 RapidShot 叠加。RapidShot 基于 DDA 的 metadata 提取光标形状和位置，不增加捕获开销。或者 v1.0 直接在受控端用 `win32gui.GetCursorInfo()` 获取位置，控制端用 CSS 渲染虚拟光标。"

---

### M1 决策

**【选定: 方案 A (DXcam) + 方案 B (WGC) 降级 + 方案 C (MSS) 兜底】**

三层降级链：
```
1. DXcam (DDA) → 首选，240FPS, <3ms
2. zbl (WGC)  → DDA 失败时，含光标捕获
3. MSS (GDI)  → 最后兜底，兼容旧系统/虚拟机
```

光标策略：Phase 1 仅传输光标坐标，控制端用 CSS 绘制虚拟光标。Phase 2 加入 RapidShot 传输真实光标纹理。

## M2: 视频编码 (Encoder)

### 方案 A: FFmpeg 子进程 — 管道喂帧

**原理:** Python `subprocess.Popen` 启动 FFmpeg，`stdin=pipe` 写入原始像素帧，FFmpeg 编码后输出到 stdout 或文件。

| 维度 | 评估 |
|------|------|
| 编码器支持 | 全部 — NVENC, AMF, QSV, libx264, libx265, AV1... |
| 延迟 (NVENC) | <5ms 编码延迟 (`-preset p1 -tune ll -delay 0`) |
| 进程管理 | 需自行处理启动/监控/重启 |
| 管道死锁风险 | 需正确管理 stdin/stderr (线程化写入) |
| 内存开销 | 低 (流式写入，不堆积帧) |
| 灵活性 | 极高 — 全 FFmpeg 参数可控 |
| 缺点 | 子进程崩溃需自行恢复；调试 FFmpeg 命令行繁琐 |

### 方案 B: 纯 Python 编码 (PyAV / av)

**原理:** 使用 `av` (PyAV) 库，在 Python 进程内直接调用 libavcodec。

| 维度 | 评估 |
|------|------|
| 硬件编码 | **不支持** NVENC/AMF/QSV 等 GPU 编码器 |
| 软件编码延迟 | 15-30ms (libx264) |
| 进程管理 | 无需，纯库调用 |
| 集成复杂度 | 低 — 同进程调用 |
| 灵活性 | 中 — FFmpeg 所有软件参数可用 |
| AI 后处理 | 方便接入 (同进程 numpy 操作) |
| 结论 | 不支持 GPU 硬编码 = 一票否决 |

### 方案 C: GStreamer 管道

**原理:** 使用 GStreamer 框架构建捕获→编码→传输管道。`dxgiscreencapsrc → nvh264enc → webrtcbin`。

| 维度 | 评估 |
|------|------|
| 硬件编码 | 全支持 (nvh264enc, amfh264enc, qsvh264enc) |
| 延迟 | 最低 — 管道路径全 GPU 驻留，理论 <1ms 编码 |
| 复杂度 | **极高** — GStreamer 在 Windows 上安装配置复杂 |
| Python 绑定 | PyGObject (需要额外安装 GTK 运行时) |
| 调试 | 困难 — 管道问题排查需要深厚 GStreamer 知识 |
| 生态 | 强在 Linux，Windows 二等公民 |
| 结论 | 性能最好但工程成本太高，不适合 Python 优先项目 |

---

### M2 辩论实录

> **架构师:** "纯 Python 编码直接毙掉 — PyAV 不支持 NVENC。这个产品最核心的竞争力就是低延迟，不用 GPU 编码就是自杀。"
>
> **PM:** "FFmpeg 子进程的方案，崩溃了怎么办？比如用户显卡驱动崩了，FFmpeg 直接挂掉。"
>
> **架构师:** "三层保障：(1) FFmpeg 心跳监控 — 主进程每 2 秒检查 FFmpeg 是否存活 (2) 崩溃自动重启 — 检测到崩溃后，重新 spawn FFmpeg，先发一个关键帧让 WebRTC 立即恢复画面 (3) 降级编码 — 如果硬件编码器反复崩溃，自动切换到 libx264 CPU 编码。但实话讲，NVENC 在驱动稳定时几乎不会崩溃。"
>
> **PM:** "GStreamer 号称延迟更低，为什么不用？"
>
> **架构师:** "因为我们是 Python 项目，不是 C++ 项目。GStreamer 在 Windows 上需要安装 MSVC 运行时、GTK 依赖、配置环境变量，用户部署成本太高。而且在 Windows 上 GStreamer 的 webrtcbin 插件稳定性不如 Linux。FFmpeg 子进程方案虽然有管道开销，但 <5ms 的编码延迟在 20-50ms 的端到端延迟预算里占比很小。"
>
> **PM:** "OK，但我还有一个担心 — 管道喂帧时如果带宽不够，FFmpeg 会不会积压导致延迟越来越大？"
>
> **架构师:** "不会。我们喂帧时如果 FFmpeg 来不及编码，它的 stdin 管道缓冲区满了，Python 的 `proc.stdin.write()` 会阻塞，天然背压。我们另一侧用 `queue.Queue(maxsize=1)` 只保留最新帧，旧帧直接丢弃。管道两端的背压机制天然防止延迟堆积。"

---

### M2 决策

**【选定: 方案 A — FFmpeg 子进程管道喂帧】**

备选保留方案 B (PyAV) 仅用于开发者环境快速搭建时的无 GPU 兜底。

编码器降级链：
```
NVENC (h264_nvenc) → AMF (h264_amf) → QSV (h264_qsv) → libx264 (CPU)
```

低延迟参数集：
```
-preset p1 -tune ll -delay 0 -rc cbr -g 120 -bf 0 -refs 1
```

心跳监控周期：2 秒。崩溃重启最大恢复时间：3 秒。

## M3: 信令与控制通道 (Network/Signaling)

### 方案对比总览

| 维度 | 方案 A: `websockets` 库 | 方案 B: `aiohttp` WebSocket | 方案 C: `python-socketio` |
|------|--------------------------|----------------------------|---------------------------|
| 定位 | 纯 WebSocket 实现 | 全功能 Web 框架 + WS | WebSocket + 长轮询降级 |
| 依赖量 | 极少 (~500KB) | 较多 (~5MB) | 中等 |
| 性能 | 最高 (专用 WS 库) | 高 | 中等 (协议开销) |
| 浏览器兼容 | 标准 WebSocket API | 标准 WebSocket API | 自动降级支持 |
| 异步支持 | 原生 asyncio | 原生 asyncio | asyncio + eventlet |
| 学习曲线 | 低 | 中 | 低 |
| 同时作为 HTTP Server | 否 (需额外 aiohttp/starlette) | **是** (一站式) | 否 |
| 消息格式 | 任意 (JSON/MsgPack/二进制) | 任意 | JSON 为主 |
| 生产成熟度 | ★★★★★ | ★★★★★ | ★★★★ |

### 编码方案对比 (控制消息序列化)

| 方案 | 编码开销 | 消息体积 | 解析速度 | 适用 |
|------|---------|---------|---------|------|
| **JSON** | 高 | 大 | 慢 | 文本数据、调试友好 |
| **MsgPack** | 低 | 小 30-50% | 快 2-3x | 二进制数据、高频事件 |
| **CBOR** | 低 | 小 | 快 | 标准化替代 |
| **自定义二进制协议** | 最低 | 最小 | 最快 | 高频键鼠事件 |

### M3 辩论实录

> **架构师:** "我建议 `websockets` 库 + aiohttp 组合。`websockets` 是纯 WebSocket 实现，性能最好，没有额外协议开销。aiohttp 只负责托管 Web 客户端静态页面。分开职责清晰。"
>
> **PM:** "为什么要分两个库？aiohttp 不是自带 WebSocket 吗？少一个依赖是一个。"
>
> **架构师:** "aiohttp 的 WebSocket 实现在高并发下有一些已知问题 — 连接管理、ping/pong 处理不如 `websockets` 库专业。我们 WebSocket 承载的是控制通道和信令通道，可靠性要求很高。`websockets` 库是 Python WebSocket 的事实标准，Starlette/Uvicorn 生态下都是用它。"
>
> **PM:** "那消息格式呢？JSON 调试方便，MsgPack 更快。"
>
> **架构师:** "混合模式。信令消息 (SDP/ICE) 用 JSON — 调试重要，且频率低。高频键鼠事件用 MsgPack 编码 — 每 5ms 一批，体积小 40%，节省带宽和序列化 CPU。剪贴板用 JSON base64 编码内容。"
>
> **PM:** "等一下，键鼠事件延迟是最影响体验的，为什么不用自定义二进制协议把体积压到最小？"
>
> **架构师:** "键盘事件结构体确实可以压到 12 字节 (4B seq + 2B type + 2B scanCode + 4B flags)，比 MsgPack 的 20-30 字节再省一半。但代价是调试困难、前后端要维护相同的协议定义、版本兼容性差。MsgPack 已经够快了，5ms 一批也就 200 条消息/秒，远不是瓶颈。v1.0 不值得为这 1ms 优化付出协议维护成本。"

---

### M3 决策

**【选定: `websockets` (信令/控制) + `aiohttp` (静态页面服务)】**

消息编码：
- 信令 (SDP/ICE): JSON
- 键鼠事件: MsgPack (高频)
- 剪贴板/文件: JSON + base64
- 系统通知: JSON

备选方案 B (aiohttp WebSocket) 保留为简配版 — 如果团队对 aiohttp 更熟悉，可以直接用 aiohttp 的 WS，基准测试差异 <5%。

## M4: WebRTC 媒体流 (WebRTC Engine)

### 方案 A: aiortc (Python 原生)

**原理:** 纯 Python 实现的 WebRTC，底层调用 PyAV (libavcodec) 进行编解码。

| 维度 | 评估 |
|------|------|
| 编码方式 | 通过 PyAV 调用 libavcodec (软编码 VP8/VP9/H.264) |
| GPU 硬编码 | 不直接支持 — 需外部 FFmpeg 编码后喂入 |
| 部署复杂度 | **低** — `pip install aiortc` 即可 |
| 延迟 | 30-80ms (软编码路径) |
| 成熟度 | 中等 — API 稳定，但小众场景可能有 bug |
| Windows 支持 | 良好 (需安装 av 的 Windows wheel) |
| 信令扩展 | 灵活 — 完全控制 SDP 生成 |
| 最大帧率 | 默认 30FPS，需修改 vpx.py 源码解除限制 |
| 多客户端 | MediaRelay 支持 (一个编码 → N 个客户端) |
| 浏览器互通 | 良好 — 标准 WebRTC 协议 |

### 方案 B: GStreamer + webrtcbin

| 维度 | 评估 |
|------|------|
| 编码方式 | GPU 硬编码 (NVENC/AMF/QSV) — 全管道 GPU 驻留 |
| 部署复杂度 | **极高** — Windows 需安装 GStreamer 运行时 + 插件 |
| 延迟 | **最低** — 10-40ms (全 GPU 管道) |
| 成熟度 | 在 Linux 上成熟，Windows 上 webrtcbin 稳定性存疑 |
| Python 绑定 | PyGObject — API 不友好 |

### 方案 C: 自定义 FFmpeg RTP 流 (无 WebRTC)

**原理:** 完全跳过 WebRTC，FFmpeg 直接输出 RTP 流，配合自定义 SDP 文件。

| 维度 | 评估 |
|------|------|
| 编码方式 | 全 GPU 硬编码 |
| 延迟 | 低 (15-40ms) |
| NAT 穿透 | **无** (LAN 场景不需要) |
| 浏览器兼容 | **差** — 需 MSE 手动喂流，延迟增加 |
| FEC/丢包恢复 | 需自行实现 |
| 信令 | 需自行设计 |
| 结论 | 太底层，工程量大，放弃 |

### 方案 D: aiortc + FFmpeg 混合编码

**核心思路:** 屏幕捕获 → FFmpeg NVENC 编码 → 将编码后的 H.264 NAL 单元包装为 aiortc 的 VideoStreamTrack。

| 维度 | 评估 |
|------|------|
| 编码延迟 | <5ms (NVENC) |
| WebRTC 兼容 | 完全标准 — 浏览器无需特殊处理 |
| 实现复杂度 | 最高 — 需自行处理 NAL 单元分包 (FU-A) 和 RTP 封装 |
| 价值 | 兼具 aiortc 的 WebRTC 生态 + NVENC 的低延迟 |

---

### M4 辩论实录

> **PM:** "我原来以为 aiortc 一把梭就行，今天才知道它不走 GPU 编码？那延迟不就上天了？"
>
> **架构师:** "对，aiortc 内置的 VP8/VP9/H.264 都是走 PyAV 软编码。1080p@60fps 软编码延迟 15-30ms。这就是为什么我坚持用 FFmpeg 做编码。"
>
> **PM:** "那方案 D 听起来既用了 WebRTC 生态又用了 NVENC，为什么不是首选？"
>
> **架构师:** "因为实现复杂度。我们需要把 FFmpeg 输出的 H.264 Annex B 码流解析成 NAL 单元，按 MTU 大小分包 (FU-A)，封装 RTP 包，再喂给 aiortc。这个工作量很大，而且很容易出 bug — 分包逻辑错一个 bit 画面就花。v1.0 我们走一个更务实的路线..."
>
> **PM:** "什么路线？"
>
> **架构师:** "**双轨并行**：(1) 默认用 FFmpeg NVENC + aiortc VideoStreamTrack 软编码兜底 — 实际上 aiortc 内置的 libx264 软编码在 LAN 上也够快 (15ms)，先跑通全链路 (2) 留出 `ExternalEncoder` 接口，Phase 2 实现方案 D 的 NVENC NAL 注入。这样 v1.0 能准时交付，v1.5 把延迟从 30ms 压到 10ms 以内。"
>
> **PM:** "等等，15ms 的软编码加上 DDA 的 3ms、网络 5ms、浏览器解码 5ms，端到端 ~30ms，这也在 30ms 目标范围内啊。"
>
> **架构师:** "对，所以方案 D 的优先级其实没有想象中高。真正的大头优化可能在浏览器解码端 — 如果浏览器启用了硬件解码 (Edge/Chrome 默认支持)，解码延迟也很低。我在想 v1.0 要不要直接上方案 D... 再给我一天做 POC 测试。"
>
> **PM:** "先写到方案里作为 v1.5 目标。v1.0 用 aiortc 内置编码把链路跑通。"

---

### M4 决策

**【选定: 方案 A (aiortc 内置编码) — v1.0 快速交付】**
**【演进: 方案 D (aiortc + FFmpeg NVENC NAL 注入) — v1.5 性能优化】**

aiortc 最大帧率补丁：修改 `aiortc/codecs/vpx.py` 中 `max_framerate = 30` → `60`。

MediaRelay 用于多客户端场景。

备选方案 B (GStreamer) 记录为长期参考 — 如果 Python 方案达到性能瓶颈，考虑用 GStreamer 重写编码管线。

## M5: 输入模拟 (Input)

### 方案 A: SendInput (Win32 API, ctypes 调用)

**原理:** 直接调用 `user32.dll` 的 `SendInput` 函数。这是 Windows 最底层的输入模拟 API。

| 维度 | 评估 |
|------|------|
| 延迟 | **<1ms** (内核级注入) |
| CPU 占用 | 极低 |
| 游戏兼容性 | 一般 — 部分 DirectInput 游戏不认 SendInput |
| UIPI 限制 | 受 UIPI (用户界面特权隔离) 限制，不能向高权限窗口注入 |
| 键码支持 | 完整 — 包括多媒体键、Unicode |
| 鼠标模式 | 绝对坐标 + 相对坐标 |
| 依赖 | 无外部库 — 纯 ctypes |
| 代码量 | ~150 行 (键盘 + 鼠标完整封装) |

### 方案 B: PyNput

**原理:** 跨平台输入库，Linux 用 X11/Xtest，Windows 用 SendInput 封装。

| 维度 | 评估 |
|------|------|
| 延迟 | <2ms (底层仍是 SendInput) |
| CPU 占用 | 低 |
| 游戏兼容性 | 同 SendInput |
| 鼠标模式 | 仅相对坐标 (无绝对坐标) |
| 跨平台 | Linux / macOS / Windows |
| 依赖 | python3-xlib (Linux), Quartz (macOS) |
| API 友好度 | 高 — `mouse.position = (x, y)` |

### 方案 C: PyDirectInput / PyAutoGUI

| 维度 | PyDirectInput | PyAutoGUI |
|------|---------------|-----------|
| 原理 | SendInput + scancode | SendInput 封装 |
| 游戏兼容 | 较好 (scancode 注入) | 差 — 大量游戏不认 |
| 延迟 | <2ms | 5-10ms (有内置 pause) |
| 依赖 | 少 | 多 (Pillow, pyscreeze...) |
| 绝对坐标 | 支持 | 支持 |

---

### M5 辩论实录

> **PM:** "PyNput 用的人最多，API 也简单，直接用不行吗？"
>
> **架构师:** "PyNput 有 3 个问题：(1) 鼠标只支持相对坐标，远程桌面需要绝对坐标 — 想象一下你点左下角的开始菜单，控制端发来 (100, 1000)，受控端需要用绝对坐标定位 (2) PyNput 在 Windows 上底层也是调 SendInput，多了一层 Python 对象包装，高频移动时 GC 压力大 (3) 依赖了 python3-xlib 即使在 Windows 上也安装，浪费。"
>
> **PM:** "那 SendInput 直接用 ctypes，游戏里能用吗？很多游戏用 DirectInput 和 RawInput。"
>
> **架构师:** "好问题。这是 SendInput 的已知局限。游戏通常用 DirectInput 直接从 HID 驱动读输入，绕过了消息队列，所以 SendInput 注入的消息它们不认。解决方案有两个：(1) 对于游戏场景，Phase 2 集成 ViGEmBus 虚拟手柄驱动，走内核驱动级注入 (2) 对于普通桌面场景，SendInput 完全够用。v1.0 的场景是远程办公/编程，不是远程打游戏。"
>
> **PM:** "OK，但 PyAutoGUI 自带了 `failSafe` 和各种安全机制，SendInput 自己写容易出错吧？"
>
> **架构师:** "安全机制我们在控制端处理，不在受控端。受控端就是忠实执行。而且 SendInput 的 ctypes 封装其实很简单 — 定义 INPUT 结构体，填好参数，扔过去。~150 行能写完的为什么要拉一个 20MB 的库。"

---

### M5 决策

**【选定: 方案 A — SendInput (ctypes 直接调用 Win32 API)】**

鼠标：绝对坐标模式 (MOUSEEVENTF_ABSOLUTE | MOUSEEVENTF_MOVE)，坐标归一化到 [0, 65535]。
键盘：scancode 注入 (KEYBDINPUT.wScan)，兼容更多应用。
滚动：MOUSEEVENTF_WHEEL，累积 delta 值。

备选方案 B (PyNput) 保留为跨平台场景方案。

v1.5 扩展：ViGEmBus 集成用于游戏手柄模拟。

### M6 辩论实录

> **PM:** "我感觉 UDP 广播就够了。我们场景就是局域网，就几台电脑，用不着 mDNS 那么重的协议。"
>
> **架构师:** "你说得对 — 对于 2-5 台电脑的典型家用/小办公室局域网，UDP 广播完全够用。但 mDNS 有一个关键好处：**Windows 以外的设备也能发现**。如果你用 iPad 的 Safari 浏览器当控制端，UDP 广播在 iOS 上不可行 — 浏览器的 UDP socket 能力受限。mDNS 则不同，iOS/macOS 原生支持 Bonjour。"
>
> **PM:** "但是 mDNS 在 Windows 上需要额外安装 Bonjour 服务，这增加了部署复杂度。"
>
> **架构师:** "对，这是痛点。我们做一个双通道发现：(1) 默认用 UDP 广播 — 最轻量，Windows→Windows 和 Windows→Android 都能用 (2) 可选启用 mDNS — 适合混合设备环境。控制端 Web 页面同时监听 WebSocket 广播响应和进行 mDNS 查询。"
>
> **PM:** "手动输入 IP 还要保留吗？"
>
> **架构师:** "必须保留 — 跨子网、VPN 接入、或广播被路由器拦截时的手动后备。而且实现起来就是 HTML 里一个 `<input>` 框，成本为零。"

---

### M6 决策

**【选定: 方案 C — UDP 广播为主 + 手动 IP 保底 + mDNS 可选】**

三层发现策略：
```
1. UDP 广播 (默认) → 2-5s 内收到响应 → 展示设备列表
2. mDNS 查询 (可选)  → 当检测到 Mac/iOS 客户端时自动启用
3. 手动输入 IP     → 始终可用，作为后备
```

UDP 广播端口：9876。受控端监听该端口，收到 `DISCOVER_SCREEN_WINDOWS` 后回复 `{"name": "...", "port": 8765, "version": "..."}`。

备选方案 A (mDNS) 保留为全功能启用时的首选。Phase 2 统一到 mDNS。

## M7: 自适应质量引擎 (Quality/Adaptive)

### 方案 A: 固定阈值五档策略

**原理:** 根据 RTT 和丢包率查表选择预设档位。

| 维度 | 评估 |
|------|------|
| 复杂度 | 低 — 查表法 |
| 鲁棒性 | 好 — 无振荡风险 |
| 档位切换 | 有阶跃感 (5 档) |
| 实现 | ~200 行 |
| 调试 | 容易 |

### 方案 B: PID 控制连续调节

**原理:** 用 PID 控制器根据 RTT 误差连续调节码率。

| 维度 | 评估 |
|------|------|
| 复杂度 | 高 — 需整定 PID 参数 |
| 平滑性 | 好 — 无级调节 |
| 鲁棒性 | 差 — 参数整定不好会振荡 |
| 参数整定 | 需要大量场景测试 |
| 结论 | 过度设计，v1.0 不值得 |

### 方案 C: Google Congestion Control (GCC) / 仅依赖 WebRTC 内置

**原理:** WebRTC 内置的 GCC 算法已经处理了拥塞控制，应用层无须重复控制。

| 维度 | 评估 |
|------|------|
| 复杂度 | 最低 — 零应用层代码 |
| 适用性 | GCC 主要基于延迟梯度，适合互联网场景 |
| LAN 问题 | LAN 延迟极低且稳定，GCC 探测不到有效的延迟梯度信号 |
| 结论 | LAN 场景下 GCC 可能过于保守，浪费带宽 |

---

### M7 辩论实录

> **架构师:** "五档策略够用了。LAN 的特点是网络状况相对稳定 — 要么有线(极好)，要么 WiFi 5G(很好)，要么 WiFi 2.4G(一般)。网络条件变化通常是离散的(从客厅走到书房)，不是连续的。五档足够覆盖。"
>
> **PM:** "PID 听起来更高科技啊，为什么不要？"
>
> **架构师:** "因为不值。PID 的收益是平滑调节，但 LAN 场景用户网络条件变化本身就是阶跃的。而且 PID 参数整定是门玄学 — Kp、Ki、Kd 三个参数，每个网络环境最优值都不一样。测试成本太高，出了问题难排查。查表法 — 出问题一眼看出是哪个档位条件不对。"
>
> **PM:** "WebRTC 内置的 GCC 不是已经有拥塞控制了？我们再搞一套不是重复？"
>
> **架构师:** "GCC 在 LAN 场景太保守了。它的设计假设是互联网 — 有瓶颈路由、缓冲膨胀、长肥管道。LAN 上延迟梯度极小 (<1ms 变化)，GCC 检测不到信号，会一直用低码率。我们的应用层 AQE 是基于端到端测量(RTT + 丢包率)，比 GCC 更直接。两者互补不冲突。"

---

### M7 决策

**【选定: 方案 A — 固定阈值五档策略】**

档位定义（从原方案细化）：

| 档位 | FPS | 分辨率 | 码率 | RTT 阈值 | 丢包率阈值 | 触发场景 |
|------|-----|--------|------|----------|------------|----------|
| 极速 | 60 | 原生 | 20M | <5ms | <0.1% | 千兆有线 |
| 高速 | 60 | 1080p | 10M | <10ms | <1% | WiFi 6 / 5G |
| 标准 | 30 | 1080p | 5M | <30ms | <3% | WiFi 5G 隔墙 |
| 节能 | 30 | 720p | 2M | <60ms | <5% | WiFi 2.4G |
| 极限 | 15 | 720p | 0.5M | >60ms | >5% | 弱信号 |

升降档策略：
- 升档条件：连续满足 5 秒（防抖动）
- 降档条件：连续触发 1.5 秒（快速响应恶化）

备选方案 C (GCC only) 作为降级 — 当 AQE 模块异常时，回退到 WebRTC 内置 GCC。

## M8: 剪贴板同步 (Clipboard)

### 方案 A: pywin32 轮询

**原理:** 受控端每 500ms 用 `win32clipboard` 检查剪贴板内容变化，计算 hash 后推送。

| 维度 | 评估 |
|------|------|
| 支持格式 | 文本、RTF、HTML、图片 (BMP/PNG)、文件列表 |
| 延迟 | 最坏 500ms |
| CPU 占用 | 极低 (仅 hash 计算) |
| 冲突处理 | 需处理"我刚同步过去的内容又被同步回来"的环路问题 |
| 实现复杂度 | 中等 |
| 依赖 | pywin32 |

### 方案 B: Windows Clipboard Listener (SetClipboardViewer)

**原理:** 注册为剪贴板查看链的成员，获得剪贴板变化通知（推送而非轮询）。

| 维度 | 评估 |
|------|------|
| 延迟 | 实时 (<10ms) |
| 可靠性 | 可能丢失 (链中其他程序没有正确传递消息) |
| 兼容性 | Win10+ 推荐用 `AddClipboardFormatListener` 替代 |
| 实现复杂度 | 高 — 需要消息循环 (Windows 窗口过程) |

### 方案 C: `AddClipboardFormatListener` + 隐藏窗口

**原理:** 创建隐藏窗口，调用 `AddClipboardFormatListener(hwnd)`，剪贴板变化时收到 `WM_CLIPBOARDUPDATE` 消息。

| 维度 | 评估 |
|------|------|
| 延迟 | 实时 |
| 可靠性 | 高 (Win10+ 推荐 API) |
| 实现复杂度 | 高 — 需要运行 Windows 消息循环 |
| 依赖 | pywin32 + ctypes |

---

### M8 辩论实录

> **架构师:** "剪贴板同步看起来简单，其实是最容易出 bug 的模块。核心问题只有一个：**环路检测**。"
>
> **PM:** "什么环路？"
>
> **架构师:** "你在受控端复制了一段文字 → 受控端检测到变化 → 推送给控制端 → 控制端收到后写入了自己的剪贴板 → 控制端的剪贴板变化监听又检测到变化 → 推送给受控端 → 受控端写入 → 受控端检测到 → 推送给控制端... 无限循环。"
>
> **PM:** "怎么解决？"
>
> **架构师:** "三个机制：(1) 写入剪贴板前，先计算内容的 SHA256 hash。写入后短暂忽略（200ms 内）同 hash 的变化通知 (2) 每一条同步消息带 `source` 字段 — 标记是 'host' 还是 'viewer'。如果接收方发现是自己的 source，忽略 (3) 维护一个最近同步记录的 LRU 缓存，如果 2 秒内看到相同内容，忽略。"
>
> **PM:** "方案选哪个？"
>
> **架构师:** "v1.0 用方案 A (pywin32 轮询) — 虽然 500ms 有延迟，但不需要隐藏窗口和消息循环，实现简单、稳定性好。而且剪贴板在远程桌面场景对延迟不敏感 — 用户 Ctrl+C 之后通常要过几秒才会在另一端 Ctrl+V。v1.5 升级到方案 C (AddClipboardFormatListener) — 实现实时推送。"

---

### M8 决策

**【选定: v1.0 — 方案 A (pywin32 轮询 500ms)；v1.5 — 方案 C (AddClipboardFormatListener 实时推送)】**

支持格式优先级：text/plain → text/html → image/png → file list。

环路检测三重机制：hash 去重 + source 标记 + LRU 时间窗口。

备选方案 B (SetClipboardViewer) 弃用 — 已被 Windows 标记为 deprecated。

## M9: 文件传输 (File Transfer)

### 方案 A: WebSocket 分片传输

**原理:** 利用已有的 WebSocket 控制通道传输文件，64KB 分片。

| 维度 | 评估 |
|------|------|
| 实现复杂度 | 低 — 复用已有 WS 连接 |
| 传输速度 | 中 — 受 WS 单通道带宽限制 (~50-100MB/s on LAN) |
| 断点续传 | 需自行实现 |
| 目录传输 | 需自行打包 |
| 并发 | 单文件串行 |
| 服务端端口 | 无需额外端口 |

### 方案 B: HTTP Range 分段传输

**原理:** 受控端启动 HTTP 文件服务，控制端用 HTTP GET 请求带 Range header 分段下载。

| 维度 | 评估 |
|------|------|
| 实现复杂度 | 低 — aiohttp 自带文件服务 |
| 传输速度 | 高 — 浏览器原生下载优化 |
| 断点续传 | HTTP Range 原生支持 |
| 并发 | 支持 (浏览器多连接) |
| 加密 | 走 HTTPS |
| 端口 | 需额外 HTTP 端口 (或复用 8766) |

### 方案 C: 专用 TCP 数据通道 (aiortc DataChannel)

**原理:** WebRTC DataChannel 基于 SCTP 协议，支持文件二进制传输。

| 维度 | 评估 |
|------|------|
| 实现复杂度 | 中 — 需管理 DataChannel 状态 |
| 传输速度 | 高 — UDP 底层，无 TCP ACK 开销 |
| 断点续传 | 需自行实现 |
| 优势 | 与视频流共用 P2P 连接 |
| 劣势 | DataChannel 大文件支持不成熟 |

---

### M9 辩论实录

> **PM:** "文件传输是远程桌面的标配功能。我看方案 B 最简单——就直接用 aiohttp 的静态文件服务，受控端把要传的文件放到一个目录，生成下载链接给控制端。浏览器原生下载还有进度条。"
>
> **架构师:** "方案 B 的问题在于只能是 Server→Client 方向。如果控制端想传文件给受控端呢？需要反过来在控制端起一个 HTTP Server。但这在我们的 B/S 架构下意味着要在浏览器里跑一个 HTTP Server — 不可行。"
>
> **PM:** "那方案 A 的 WebSocket 双向都可以传。"
>
> **架构师:** "对，而且 WebSocket 我们已经有现成的连接，不需要额外端口和安全配置。64KB 分片+ACK 确认+重传机制，实现一个简单的可靠文件传输协议就够了。LAN 下带宽不是瓶颈，重点是可靠性和断点续传。"
>
> **PM:** "方案 C 的 DataChannel 呢？"
>
> **架构师:** "DataChannel 基于 SCTP over UDP，是最优雅的方案 — 和视频流共用 P2P 通道，没有额外连接。但是 aiortc 的 DataChannel 实现在大文件场景（>100MB）有已知的内存问题，而且断点续传得自己写。v1.0 不建议踩坑。"

---

### M9 决策

**【选定: 方案 A — WebSocket 分片传输】**

协议设计：
```
Client→Host: {"type": "file_req", "id": "uuid", "action": "send", "name": "...", "size": 12345}
Host→Client: {"type": "file_req", "id": "uuid", "status": "ready"}
Client→Host: {"type": "file_chunk", "id": "uuid", "offset": 0, "data": "<base64>"}
Host→Client: {"type": "file_chunk_ack", "id": "uuid", "offset": 65536}
...
Client→Host: {"type": "file_req", "id": "uuid", "action": "complete"}
```

分片大小：64KB。每 10 片等待一次批量 ACK（减少 ACK 开销）。
超时重传：500ms 未收到 ACK 则重发。
断点续传：记录已确认的最大连续 offset，重连后从断点继续。

## M10: 认证与会话 (Auth)

### 方案 A: PIN 码 + Token

**原理:** 首次配对时输入 6 位 PIN 码，验证通过后生成长效 Token。后续连接使用 Token 自动认证。

| 维度 | 评估 |
|------|------|
| 安全性 | 中 (Token 本地存储，PIN 仅首次交互) |
| 用户体验 | 好 (首次配对后自动连接) |
| 实现复杂度 | 低 |
| 适用场景 | 局域网，设备可信 |

### 方案 B: 密码认证 (每次输入)

| 维度 | 评估 |
|------|------|
| 安全性 | 高 |
| 用户体验 | 差 (每次连接都要输) |
| 适用场景 | 安全敏感环境 |

### 方案 C: 证书认证 (TLS 双向)

| 维度 | 评估 |
|------|------|
| 安全性 | 最高 |
| 用户体验 | 差 — 需分发和管理证书 |
| 适用场景 | 企业级 |
| 结论 | 过度设计，LAN 场景不需要 |

---

### M10 决策

**【选定: 方案 A — PIN + Token，可选"每次询问"模式】**

```
首次配对流程:
  控制端 ──[发现受控端]──→ 受控端
  控制端 ──[请求配对]──→ 受控端
  受控端 显示 6 位 PIN (或用户自定义)
  控制端 用户在 Web 页面输入 PIN
  控制端 ──[PIN + client_public_key]──→ 受控端
  受控端 验证 PIN，生成 session_token，用 client_public_key 加密返回
  双方 保存 token 到本地 (JSON 文件，权限 0600)

后续连接:
  控制端 ──[token + timestamp + HMAC]──→ 受控端
  受控端 验证 HMAC 和时间戳 (5 分钟窗口)
```

备选方案 B (每次密码) 作为可配置选项。

## M11: 性能监控与统计 (Stats)

### 方案 A: 自定义 Stats Collector

**原理:** 各模块上报指标到统一的 StatsCollector，控制端每 2 秒拉取一次。

| 维度 | 评估 |
|------|------|
| 指标覆盖 | 完全可控 |
| 实现复杂度 | 中 |
| 调试支持 | 日志输出 + Web UI 面板 |

### 方案 B: WebRTC getStats API

**原理:** 从 RTCPeerConnection.getStats() 获取编解码/网络统计。

| 维度 | 评估 |
|------|------|
| 指标覆盖 | 仅 WebRTC 内部指标 (编解码、丢包、RTT、带宽) |
| 实现复杂度 | 低 — aiortc 内置支持 |
| 调试支持 | 丰富 — 标准化指标 |

### 方案 C: Prometheus + Grafana

**原理:** 受控端暴露 Prometheus metrics endpoint，控制端用 Grafana 面板。

| 维度 | 评估 |
|------|------|
| 指标覆盖 | 完全覆盖 |
| 实现复杂度 | 高 — 需要 Prometheus + Grafana 部署 |
| 适用场景 | 生产监控 |
| 结论 | 过度设计 — v1.0 目标是客户端调试面板 |

---

### M11 决策

**【选定: 方案 A + 方案 B 混合】**

- WebRTC 层指标: 使用 `getStats()` 获取 (丢包率、RTT、编解码延迟、带宽估计)
- 应用层指标: 自定义 collector (捕获帧率、编码前帧率、WS 消息延迟、内存/CPU)
- 展示: Web 客户端内置 Stats 面板 (半透明覆盖，快捷键 Ctrl+Shift+S 切换)

关键监控指标：
```

[Encoder]  FPS: 60 | Latency: 4.1ms | Codec: h264_nvenc
[WebRTC]   RTT: 3ms | Loss: 0.02% | Bitrate: 8.2 Mbps | NACK: 0
[WS Ctrl]  RTT: 2ms | Queue: 0
[System]   CPU: 3.2% | MEM: 142MB
```

## M12: Web 客户端 (Web UI)

### 方案 A: 纯 HTML/CSS/JS (无框架)

**原理:** 单页面应用，原生 DOM API + WebRTC API + WebSocket API。

| 维度 | 评估 |
|------|------|
| 体积 | 极小 (<50KB gzipped) |
| 加载速度 | 即时 |
| 依赖 | 零 |
| 浏览器兼容 | Chrome/Edge/Firefox/Safari |
| 开发效率 | 低 (手写 DOM 操作) |
| 维护性 | 中等 |
| 打包 | 不需要 |

### 方案 B: Vue/React SPA

| 维度 | 评估 |
|------|------|
| 体积 | 大 (200KB+ gzipped) |
| 加载速度 | 慢 (JS bundle 解析) |
| 依赖 | node_modules 地狱 |
| 开发效率 | 高 |
| 维护性 | 好 |

### 方案 C: htmx + 原生 WebRTC

| 维度 | 评估 |
|------|------|
| 体积 | 小 (~14KB) |
| 开发效率 | 中 |
| 适用性 | htmx 适合 CRUD，不适合实时视频 |
| 结论 | 主要 UI 是 video 标签，htmx 帮不上忙 |

---

### M12 辩论实录

> **PM:** "用 React 吧，团队熟悉，组件化开发效率高。"
>
> **架构师:** "React 打包后 ~40KB(gzipped)，再加 react-dom 130KB。但我们的 Web 客户端 UI 只有 4 个部分：(1) 连接面板 (2) 视频画面 (3) 工具栏 (4) 设置对话框。没有路由、没有状态管理复杂需求、没有列表渲染。React 的收益几乎为零。"
>
> **PM:** "那至少用个轻量框架？"
>
> **架构师:** "直接用原生 JS。整个客户端不超过 500 行 JS。视频就是 `<video>` 标签设置 `srcObject`，工具栏几个图标按钮，设置一个 `<dialog>`。用框架反而要处理构建工具链、HMR、tree-shaking，增加工程复杂度。而且我们的 HTML 是由 aiohttp 直接 serve 的，不需要 webpack 打包。"
>
> **PM:** "样式呢？"
>
> **架构师:** "CSS Variables + Flexbox + Grid。再加一个 system font stack 搞定字体。不需要 Tailwind，不需要 CSS-in-JS。目标是受控端 pip install 后，控制端浏览器输入 `http://<host>:8766` 立即能用。"

---

### M12 决策

**【选定: 方案 A — 纯 HTML/CSS/JS 零依赖单页面】**

页面组成：
```
┌─────────────────────────────────┐
│  [Logo]    设备名称    [设置 ⚙] │ ← 顶栏 48px
├─────────────────────────────────┤
│                                 │
│                                 │
│         <video> 画面区           │ ← 主体区域
│         自适应填充               │
│                                 │
│                                 │
├─────────────────────────────────┤
│  [全屏] [质量] [剪贴板] [文件]  │ ← 工具栏 40px
└─────────────────────────────────┘
```

键盘/鼠标事件处理：控制端捕获 DOM 事件 → 聚合 → MsgPack 编码 → WebSocket 发送。
全屏模式：Pointer Lock API 锁定鼠标，隐藏工具栏。
移动端适配：触控映射 (单指移动=鼠标移动, 单指点按=左键, 双指=右键, 双指捏合=滚轮)。

## M13: 音频捕获 (Audio — Phase 2)

### 方案 A: PyAudio + WASAPI Loopback

**原理:** 通过 WASAPI loopback 设备捕获系统音频输出。

| 维度 | 评估 |
|------|------|
| 延迟 | ~10-20ms |
| 设备选择 | 需枚举并选择正确的输出设备 |
| 独占模式 | 可获取更低的延迟 (exclusive mode) |
| 依赖 | PyAudio (PortAudio 绑定) |

### 方案 B: FFmpeg WASAPI 捕获

**原理:** FFmpeg `-f dshow -i audio="Stereo Mix"` 直接捕获并编码。

| 维度 | 评估 |
|------|------|
| 延迟 | ~15-30ms |
| 集成度 | 与视频 FFmpeg 共享子进程管理 |
| 限制 | Stereo Mix 在很多设备上默认禁用 |

---

### M13 决策

**【Phase 2 引入，推荐方案 A (PyAudio WASAPI)】**

编码方案: Opus (WebRTC 首选音频编码)，通过 aiortc AudioStreamTrack 传输。
延迟容忍: 音频比视频延迟高 20ms 是可接受的 (人耳对 <50ms A/V 异步不敏感)。

## M14: 游戏手柄模拟 (Gamepad — Phase 2)

### 方案 A: ViGEmBus 驱动

**原理:** 安装虚拟 Xbox 360 / DS4 手柄驱动，通过 ViGEmClient.dll 模拟。

| 维度 | 评估 |
|------|------|
| 兼容性 | 极高 — 系统识别为标准 Xbox 手柄 |
| 游戏支持 | 所有支持手柄的游戏 |
| 部署 | 需安装驱动 (管理员权限) |
| Python 绑定 | 通过 ctypes 调用 ViGEmClient.dll |

### 方案 B: vJoy

**原理:** 虚拟手柄驱动，更灵活但更复杂。

| 维度 | 评估 |
|------|------|
| 灵活性 | 高 — 可自定义轴数/按钮数 |
| 部署复杂度 | 高 — 配置复杂 |
| 推荐度 | 不如 ViGEmBus |

---

### M14 决策

**【Phase 2，推荐方案 A (ViGEmBus)，仅 v1.5 原生客户端支持】**

Web 客户端受 Gamepad API 限制，浏览器仅能读取物理手柄，无法转发。Phase 2 PySide6 原生客户端支持。

```toml
# ~/.screen_windows/host_config.toml

[server]
port = 8765
http_port = 8766
bind = "0.0.0.0"

[capture]
backend = "dxcam"          # dxcam | wgc | mss
monitor = 0                # 0 = primary
max_fps = 60
capture_cursor = false     # Phase 2

[encoder]
backend = "auto"           # auto | nvenc | amf | qsv | libx264
codec = "h264"             # h264 | h265 | av1
bitrate = "10M"            # or "auto" for AQE
preset = "p1"

[auth]
mode = "pin_once"          # pin_once | always | none
pin = ""                   # empty = auto-generate

[discovery]
method = "udp"             # udp | mdns | both
udp_port = 9876

[advanced]
stats_port = 0             # 0 = disabled
log_level = "info"         # debug | info | warning | error
```

配置优先级：命令行参数 > 环境变量 > 配置文件 > 默认值。

备选方案 A (JSON) 保留为兼容格式 — 同时支持 JSON 和 TOML。

## 模块间数据流时序

```
受控端                                    控制端(浏览器)
──────                                    ──────

[Capture Thread]                          [Render Loop]
      │                                        ▲
      │ numpy frame (240Hz)                    │ VideoFrame
      ▼                                        │
[Frame Queue (maxsize=1)]                      │
      │                                        │
      │ latest frame (60Hz pickup)             │
      ▼                                        │
[Preprocess]                                  │
  · resize (if needed)                        │
  · bgr24 → yuv420p (if needed)               │
      │                                        │
      ▼                                        │
[aiortc VideoStreamTrack.recv()]              │
      │                                        │
      │ av.VideoFrame                          │
      ▼                                        │
[aiortc RTCPeerConnection] ──── WebRTC UDP ────► [Browser RTCPeerConnection]
      ▲                                        │
      │                                        ▼
[WebSocket Server] ◄──── Control events ──── [WebSocket Client]
      │                                        │
      ▼                                        │
[Input Executor]                              │
  · ctypes SendInput                          │
  · win32clipboard                            │
```

**关键数据流设计原则：**
1. Capture 线程以显示器刷新率运行 (60/120/144Hz)，但送入限制为 max_fps
2. FrameQueue 为 `maxsize=1` — 永远只保留最新帧，消费慢时丢弃旧帧
3. `recv()` 非阻塞 — 从 Queue 取帧，取不到返回重复上一帧或黑帧
4. WebSocket 控制通道与 WebRTC 媒体通道完全独立，互不阻塞
5. WebRTC 内部处理 UDP 丢包恢复 (NACK/FEC/RTX)，应用层不需要操心
